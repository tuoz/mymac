# macOS 自定义快捷键工具软件需求文档

## 1. 文档信息

- 文档类型：软件需求文档（SRS / PRD 合并版）
- 项目阶段：MVP 规划
- 目标平台：macOS
- 当前结论日期：2026-04-07

## 2. 项目背景

本项目旨在开发一个 macOS 常驻后台工具软件，核心能力是在全局范围内监听用户定义的快捷键，并将其映射为系统方向键行为。

当前第一阶段需求较为聚焦：

1. 支持开机自启动
2. 注册全局快捷键 `Fn + H/J/K/L`
3. 将 `Fn + H/J/K/L` 映射为 `Left/Down/Up/Right`
4. 保留其他组合修饰键，使其可以自然叠加，例如：
   - `Fn + Cmd + H -> Cmd + Left`
   - `Fn + Shift + L -> Shift + Right`

后续产品可能扩展到窗口管理、布局控制、窗口吸附、分屏快捷操作等能力，但本阶段不实现窗口管理功能，仅在架构上为后续扩展预留能力。

## 3. 已确认决策

- 软件形态：`纯后台常驻，无 Dock 图标`
- 自启动策略：`首次安装后引导用户开启，不默认强制开启`
- 快捷键语义：`保留除 Fn 外的其他修饰键，并仅将 H/J/K/L 转换为方向键`
- 发布方式：`当前以个人或团队内部使用为主`

## 4. 产品目标

### 4.1 核心目标

- 让用户在不离开键盘主区的情况下，用 `Fn + H/J/K/L` 实现方向键操作
- 使映射在大多数日常应用中保持稳定、低延迟、可预期
- 提供简单清晰的开机启动与权限引导流程
- 采用可扩展架构，为未来增加窗口管理动作打基础

### 4.2 非目标

- 本阶段不支持用户自定义任意按键映射
- 本阶段不支持窗口管理动作
- 本阶段不支持复杂配置同步、云端配置或脚本系统
- 本阶段不追求 Mac App Store 上架兼容性

## 5. 目标用户

当前目标用户为：

- 习惯 Vim 风格移动键位的 macOS 用户
- 对键盘效率敏感的开发者、设计师、重度键盘用户
- 愿意授予系统辅助功能权限、接受后台常驻工具的软件使用者

## 6. 使用场景

### 6.1 核心场景

- 用户在任何支持方向键的应用中，通过 `Fn + H/J/K/L` 进行光标移动
- 用户在文本编辑、代码编辑、表格导航、浏览器表单、终端中复用该组合
- 用户配合 `Cmd`、`Shift`、`Option` 等修饰键完成选择、跳词、切换等操作

### 6.2 典型示例

- `Fn + H` 等价于 `Left`
- `Fn + J` 等价于 `Down`
- `Fn + K` 等价于 `Up`
- `Fn + L` 等价于 `Right`
- `Fn + Cmd + H` 等价于 `Cmd + Left`
- `Fn + Shift + L` 等价于 `Shift + Right`
- `Fn + Option + H` 等价于 `Option + Left`
- `Fn + Cmd + Shift + L` 等价于 `Cmd + Shift + Right`

## 7. 产品形态与交互原则

### 7.1 产品形态

本软件采用后台常驻工具形态：

- 启动后常驻后台
- 默认无 Dock 图标
- 不要求主窗口常驻展示
- 允许提供轻量设置界面与权限引导界面

### 7.2 推荐交互入口

虽然是无 Dock 图标后台应用，但建议保留以下入口之一：

- 菜单栏图标
- 首次启动自动展示的设置窗口
- 二者兼有

本阶段推荐采用：

- 后台应用 + 菜单栏入口 + 设置窗口

原因：

- 便于查看当前运行状态
- 便于触发权限引导
- 便于切换“开机启动”选项
- 便于后续增加快捷键配置与功能扩展

## 8. 功能范围

### 8.1 MVP 功能清单

1. 首次启动引导
2. 开机自启动开关
3. 全局快捷键监听
4. H/J/K/L 到方向键的映射执行
5. 修饰键透传
6. 运行状态可视化
7. 基础错误提示与权限状态提示

### 8.2 后续预留能力

1. 自定义快捷键配置
2. 自定义动作系统
3. 窗口管理动作
4. 应用级规则
5. 配置导入导出

## 9. 功能需求

### 9.1 首次启动引导

系统首次启动时，软件应：

- 检测运行所需权限状态
- 向用户解释软件用途
- 引导用户开启必要系统权限
- 询问是否启用开机自启动
- 在用户完成或跳过后进入后台常驻模式

首次引导中至少应说明：

- 为什么需要监听全局按键
- 为什么需要辅助功能（Accessibility）权限
- 如何在系统设置中完成授权
- 未授权时会导致哪些功能不可用

### 9.2 开机自启动

系统应支持用户通过应用内开关启用或关闭开机自启动。

具体要求：

- 首次安装后默认不直接启用
- 首次运行时主动引导用户启用
- 启用后，下次登录系统时应用自动启动
- 应用应能正确读取当前自启动状态并展示给用户
- 启用失败时应给出清晰错误提示和处理建议

### 9.3 全局快捷键监听

系统应在全局范围内监听用户键盘事件，并识别以下触发条件：

- `Fn + H`
- `Fn + J`
- `Fn + K`
- `Fn + L`
- `Fn + Cmd + H/J/K/L`
- `Fn + Shift + H/J/K/L`
- `Fn + Option + H/J/K/L`
- 以及上述修饰键组合的合理叠加

约束要求：

- 仅在按下 `Fn` 且目标键为 `H/J/K/L` 时触发映射
- 对非目标按键不做处理
- 不应破坏其他普通键盘输入行为

### 9.4 键位映射规则

映射规则如下：

| 输入 | 输出 |
|---|---|
| `Fn + H` | `Left` |
| `Fn + J` | `Down` |
| `Fn + K` | `Up` |
| `Fn + L` | `Right` |

修饰键透传规则：

- 输入中的 `Cmd`、`Shift`、`Option`、`Control` 应尽可能原样附着到目标方向键事件上
- `Fn` 仅作为触发条件，不作为输出事件修饰键保留

示例：

| 输入 | 预期输出 |
|---|---|
| `Fn + Cmd + H` | `Cmd + Left` |
| `Fn + Shift + L` | `Shift + Right` |
| `Fn + Option + H` | `Option + Left` |
| `Fn + Cmd + Shift + K` | `Cmd + Shift + Up` |

### 9.5 按键处理行为

系统对命中映射规则的按键事件应执行以下行为：

1. 拦截原始 `H/J/K/L` 输入
2. 生成对应方向键事件
3. 将保留的修饰键附加到新事件中
4. 分发到当前前台应用

系统应关注以下体验要求：

- 尽量降低输入延迟
- 避免按键重复、漏触发、双触发
- 长按时行为尽量接近系统原生方向键重复逻辑

### 9.6 权限状态管理

系统应能感知并展示关键权限状态。

至少包括：

- 是否具备执行当前键位映射链路所需的辅助功能权限
- 当权限缺失时，是否可正常工作

当权限缺失时：

- 应明确告知功能不可用
- 应提供打开系统设置的引导
- 应允许用户在授权完成后重新检测状态

### 9.7 运行状态展示

系统应至少提供一个轻量状态入口，用于展示：

- 当前是否运行中
- 快捷键映射是否启用
- 权限是否完整
- 开机启动是否已开启

建议状态文案：

- 运行正常
- 缺少权限
- 自启动未开启
- 映射暂停

### 9.8 设置项

MVP 设置项建议包含：

1. 开机自启动开关
2. 快捷键映射总开关
3. 权限状态查看与重新检测
4. 打开系统设置快捷入口
5. 关于页或版本信息

本阶段不要求：

- 自定义热键
- 自定义映射动作
- 多配置方案切换

## 10. 非功能需求

### 10.1 性能

- 快捷键触发到事件分发应尽可能低延迟，主观体验接近原生方向键
- 后台常驻状态下 CPU 与内存占用应保持较低
- 不应因轮询或不必要的监听导致系统资源浪费

### 10.2 稳定性

- 应用在权限缺失、前台应用切换、睡眠唤醒后应保持可恢复
- 当事件监听失败时，应用不应崩溃
- 启动失败、自启动注册失败、权限检测失败时应可恢复并提示用户

### 10.3 可维护性

- 模块间边界清晰
- 键位规则与动作执行逻辑解耦
- 权限引导、自启动、事件监听、动作执行应独立实现
- 新增窗口管理动作时，不应重写底层事件识别与路由机制

### 10.4 可观测性

- 应保留基础日志能力，用于记录权限变化、监听启动状态、映射命中、错误信息
- 日志默认应避免记录敏感用户输入内容

### 10.5 可用性

- 首次引导文案应通俗
- 设置项应足够少，避免配置负担
- 权限缺失时的提示应清晰直达，不让用户猜测问题原因

## 11. 权限与系统约束

由于本软件需要在全局范围监听按键并发送目标键盘事件，预期会涉及 macOS 的系统安全能力限制。

需求层面应明确：

- 软件需要申请并提示用户授予相关系统权限
- 若权限未开启，核心功能将不可用或部分不可用
- 应用需对权限状态做前置检测和持续提示

结合当前使用目标为内部使用，建议默认采用：

- 直接分发
- 代码签名
- 公证（notarization）

说明：

- 本软件的技术路径与系统权限模型更适合内部直装或官网分发
- 本阶段不以 Mac App Store 兼容性为约束

## 12. 架构需求

### 12.1 总体原则

架构应以“输入事件识别”和“动作执行”解耦为核心原则。

建议采用分层结构：

1. App Shell 层
2. 系统能力层
3. 输入映射层
4. 动作执行层
5. 配置与状态层

### 12.2 推荐模块划分

#### A. App Shell

职责：

- 应用生命周期管理
- 后台运行与菜单栏集成
- 设置界面与首次引导

#### B. Permissions Service

职责：

- 检测系统权限状态
- 提供权限申请引导
- 监听权限状态变化

#### C. LaunchAtLogin Service

职责：

- 开机自启动注册/取消注册
- 查询当前自启动状态
- 向 UI 提供状态更新

#### D. Input Capture Service

职责：

- 监听全局键盘事件
- 标准化事件结构
- 将事件上送到映射层

#### E. Key Mapping Engine

职责：

- 根据规则匹配 `Fn + H/J/K/L`
- 将输入事件转换为抽象动作
- 透传保留修饰键

建议输出不要直接写死为“发送方向键”，而是输出统一动作模型，例如：

- `sendKey(.left, modifiers: [.command])`
- `sendKey(.up, modifiers: [.shift, .option])`

#### F. Action Dispatcher

职责：

- 接收抽象动作
- 分发到具体执行器

当前 MVP 只需一个执行器：

- Keyboard Event Executor

后续可扩展执行器：

- Window Management Executor
- App Command Executor
- Script Executor

#### G. Settings Store

职责：

- 保存用户开关状态
- 保存未来的映射规则配置
- 管理版本迁移

### 12.3 扩展性要求

为了后续支持窗口管理，当前架构必须满足：

1. 快捷键识别与动作执行分离
2. 动作模型抽象化，而非仅面向方向键硬编码
3. 映射规则应支持从“键位 -> 动作描述”演进
4. 执行器应支持按动作类型分派
5. 设置存储结构应支持未来增加多条规则

建议未来目标模型示例：

- 规则：`Fn + H -> sendKey(Left)`
- 规则：`Fn + Cmd + Return -> window.snapLeft`
- 规则：`Fn + U -> window.maximize`

即本阶段虽然只实现方向键映射，但设计上应把“方向键映射”视为一种具体动作类型，而不是唯一功能。

## 13. 数据与配置需求

本阶段数据需求简单，建议至少保存以下配置：

- 使用轻量本地存储

- 后续可平滑扩展为规则列表结构

## 15. 错误处理需求

系统至少应覆盖以下异常场景：

1. 用户未授予所需权限
2. 自启动注册失败
3. 全局监听初始化失败
4. 按键事件注入失败
5. 应用启动后运行状态异常

处理原则：

- 优先提示用户可执行的修复动作
- 避免只显示技术报错
- 错误状态可恢复、可重试

## 16. 测试与验收要求

### 16.1 功能验收

以下场景通过后可认为 MVP 核心可用：

1. 用户首次启动后可以看到权限与自启动引导
2. 用户启用开机自启动后，重新登录系统应用会自动启动
3. 在常见应用中，`Fn + H/J/K/L` 能正确映射到方向键
4. `Fn + Cmd + H`、`Fn + Shift + L` 等组合可正确保留修饰键
5. 未授权时应用能给出清晰提示，授权后可恢复工作
6. 应用后台常驻，不显示 Dock 图标

### 16.2 兼容性测试建议

建议至少验证以下应用类型：

- 文本编辑器
- 代码编辑器
- 浏览器输入框
- Terminal/iTerm
- Finder 列表或文件浏览视图

### 16.3 稳定性测试建议

- 连续长时间后台运行
- 系统锁屏/解锁
- 睡眠/唤醒
- 多次切换权限状态
- 长按重复按键

## 18. 风险与注意事项

### 18.1 系统权限风险

macOS 对全局键盘监听和事件注入有明确安全限制，权限引导与状态管理将直接影响软件可用性。

### 18.2 兼容性风险

不同应用对模拟键盘事件、长按重复、组合键解释可能存在差异，需要通过实际应用场景验证。

### 18.3 后续功能演进风险

如果当前版本将“按键识别”和“方向键发送”写死在一起，未来加入窗口管理时会产生明显重构成本，因此必须在 MVP 阶段就完成动作抽象。

## 19. 当前建议的技术方向

基于当前需求，建议产品技术路线为：

- 原生 macOS 应用
- Swift 6+
- SwiftUI 负责设置与引导界面
- 必要时用 AppKit/底层系统 API 处理全局事件监听与事件注入
- 使用系统提供的登录项能力实现开机自启动
- 以“后台应用 + 菜单栏入口 + 设置窗口”作为推荐形态

## 20. 待后续确认事项

以下内容当前不影响需求文档成立，但建议在设计或开发前补齐：

1. 最低支持的 macOS 版本
2. 应用名称、Bundle ID、签名身份
3. 菜单栏是否始终显示图标
4. 首次引导是否需要多步骤说明页
5. 是否需要“临时暂停映射”快捷开关
6. 后续窗口管理功能的优先级与动作范围

## 21. 结论

本项目的 MVP 应被定义为一个面向内部使用的 macOS 后台常驻工具：

- 具备权限引导
- 支持用户启用开机自启动
- 能稳定地将 `Fn + H/J/K/L` 映射为方向键
- 能保留 `Cmd/Shift/Option/Control` 等其他修饰键
- 在架构上为未来窗口管理预留动作抽象与执行器扩展能力

这意味着第一版虽然功能简单，但在工程设计上不能把它做成“一段硬编码键盘 Hook”，而应该从一开始就将其设计为“输入规则驱动的动作执行器”。
